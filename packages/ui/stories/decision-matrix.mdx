# Component Decision Matrix

When two Kanso components solve overlapping problems, here's how to pick.
Each row points to the deciding question — read the right column first,
then the left.

## Selection inputs

<table className="kp-dm-table">
  <thead>
    <tr><th>Picking between</th><th>Pick A when…</th><th>Pick B when…</th></tr>
  </thead>
  <tbody>
    <tr><td><strong>Select</strong> vs <strong>Combobox</strong></td><td>The list is short (≤ 20 items) and the user just picks one</td><td>The list is long, async, or needs free-text search</td></tr>
    <tr><td><strong>Combobox</strong> vs <strong>CommandPalette</strong></td><td>The control is part of a form and the picked value is the field's <code>value</code></td><td>The control is global / app-level and the picked value triggers an <em>action</em></td></tr>
    <tr><td><strong>Radio</strong> vs <strong>SegmentedControl</strong></td><td>The set is mutually exclusive <em>and</em> expected to live in a form</td><td>The set is mutually exclusive <em>and</em> feels like a tab-style mode toggle</td></tr>
    <tr><td><strong>Checkbox</strong> vs <strong>Toggle</strong></td><td>The state is a form value that gets submitted on save</td><td>The state is a setting that takes effect immediately</td></tr>
    <tr><td><strong>Select</strong> vs <strong>Tree</strong></td><td>The data is flat</td><td>The data is hierarchical and depth matters to the user</td></tr>
    <tr><td><strong>DatePicker</strong> (<code>mode="single"</code>) vs <strong>DatePicker</strong> (<code>mode="range"</code>)</td><td>One date is enough (birthday, deadline)</td><td>The user picks a span (booking, report range)</td></tr>
  </tbody>
</table>

## Notification surfaces — full hierarchy

When you have a "tell the user something" job, the question is mostly
about *scope* and *persistence*:

```
                    GLOBAL                 PER-INTERACTION
                    ──────                 ───────────────
PERSISTENT      Banner                 Alert
TRANSIENT       Toast (corner)         FormField error / Tooltip
```

- **Banner** — top-of-page, applies to the whole session ("You're in
  read-only mode").
- **Alert** — sits inside content, applies to a specific area ("This
  workspace has 47 errors").
- **Toast** — corner, fades on its own, confirms an action ("Saved").
- **FormField error** — inline below a field, scoped to that input.
- **Tooltip** — explains a single trigger on hover/focus, never errors.

## When in doubt

- The component you're considering doesn't fit any row above? Start with
  the per-component doc — every page in [`docs/components/`](https://github.com/GregNBlack/kanso-protocol/tree/main/docs/components)
  has a "Do / Don't" section that handles edge cases.
- Two components could *both* work? Pick the one with the more
  conservative API surface (fewer inputs, smaller bundle, marked
  `stable` in `docs/stability.md`).
- Still stuck? Open a discussion on GitHub — questions like "X vs Y"
  often surface real ambiguity that this matrix should be updated to
  cover.
