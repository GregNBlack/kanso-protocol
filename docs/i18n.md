# Internationalization (i18n)

Kanso ships a small, opt-in i18n surface. Two injection tokens + a handful of `Intl`-backed helpers cover the common cases (date / time formatting, weekday names, button labels) without dragging in a translation framework.

The package is `@kanso-protocol/i18n`. Components that need locale-aware behavior depend on it as a peer; consumers provide the tokens at the app root.

## The two tokens

### `KP_LOCALE` — BCP-47 language tag

Drives every Intl-backed call: month names, weekday labels, time formatting, AM/PM markers.

Default factory: `navigator.language` if available, else `'en-US'`.

```ts
import { KP_LOCALE } from '@kanso-protocol/i18n';

bootstrapApplication(AppComponent, {
  providers: [
    { provide: KP_LOCALE, useValue: 'fr-FR' },
  ],
});
```

### `KP_STRINGS` — static UI labels

A `Partial<KpLocaleStrings>` registry. Override only the keys you need; missing keys fall through to English defaults.

```ts
import { KP_STRINGS } from '@kanso-protocol/i18n';

bootstrapApplication(AppComponent, {
  providers: [
    {
      provide: KP_STRINGS,
      useValue: {
        close: 'Закрыть',
        clear: 'Очистить',
        today: 'Сегодня',
        yesterday: 'Вчера',
        thisMonth: 'Этот месяц',
        lastMonth: 'Прошлый месяц',
        commandPalettePlaceholder: 'Команда или поиск…',
      },
    },
  ],
});
```

The full key list lives in `KpLocaleStrings` (see [tokens.ts](https://github.com/GregNBlack/kanso-protocol/blob/main/packages/i18n/src/tokens.ts)). The interface stays small on purpose — anything Intl can derive (month names, day periods, time formats) is **not** in the registry.

## The Intl helpers

Use these inside components instead of hand-rolling `Intl.DateTimeFormat`. They cache formatters per `(locale, options)` pair so calling them in a render loop is cheap.

```ts
import {
  kpFormatDate,
  kpFormatTime,
  kpMonthNames,
  kpWeekdayNames,
  kpDayPeriod,
  injectKpLocale,
} from '@kanso-protocol/i18n';

const locale = injectKpLocale();

kpFormatDate(new Date(), locale, { dateStyle: 'medium' });  // "May 4, 2026"
kpFormatTime(new Date(), locale);                            // "3:45 PM" or "15:45"
kpMonthNames(locale, 'long');                                // ['January', ..., 'December']
kpWeekdayNames(locale, 1, 'narrow');                         // ['M', 'T', 'W', 'T', 'F', 'S', 'S']
kpDayPeriod(15, 'en-US');                                    // 'PM' (or '' for 24h locales)
```

## Usage in a Kanso component

```ts
import { Component, inject } from '@angular/core';
import { injectKpLocale, injectKpStrings, kpFormatDate } from '@kanso-protocol/i18n';

@Component({
  template: `
    <button [attr.aria-label]="strings.clear">×</button>
    <span>{{ today }}</span>
  `,
})
export class MyComponent {
  private locale = injectKpLocale();
  protected strings = injectKpStrings();
  protected today = kpFormatDate(new Date(), this.locale);
}
```

| Component | Status | What's localized |
|-----------|--------|------------------|
| `datepicker` | ✓ | Month names (long + short), weekday labels, date format, preset labels (Today / Yesterday / This week / etc.), ARIA labels for clear / prev / next |
| `form-field` | ✓ (since `0.1.0`) | Validation messages via the older `KP_VALIDATION_MESSAGES` token (predates this package — will be folded into `KP_STRINGS` in a future minor) |
| `timepicker` | ✗ TBD | 12h/24h split needs Intl-driven format detection |
| `file-upload` | ✗ TBD | `dropzoneTitle`, `dropzoneRemove(filename)`, `dropzoneUploaded` exist in `KP_STRINGS` but the component still has them as `@Input` defaults |
| `command-palette` | ✗ TBD | `commandPalettePlaceholder`, `noResults`, `commandPaletteHint` exist in `KP_STRINGS`, awaiting wiring |
| `search-bar` | ✗ TBD | Same as above for `searchPlaceholder` |

## Why not full message catalogs?

A full message-catalog framework (Angular i18n, ngx-translate, transloco) is the right tool when you have hundreds of strings and a translation pipeline. For a design system where each component owns ~5 strings and consumers usually translate at the app level anyway, a small DI-backed registry is enough.

If a Kanso consumer wants to plug into a real catalog, they can — wrap their catalog's lookup in a `Partial<KpLocaleStrings>` factory and provide it via `KP_STRINGS`.

## Future work

- Migrate timepicker (12h/24h via `kpDayPeriod()`).
- Migrate file-upload, command-palette, search-bar to `KP_STRINGS`.
- Fold `KP_VALIDATION_MESSAGES` into `KP_STRINGS` as a sub-namespace (probably `validation.required`, `validation.minlength`, etc.) — without breaking the existing token.
- Number formatting helpers (`kpFormatNumber`, `kpFormatCurrency`) when a component actually needs them.
