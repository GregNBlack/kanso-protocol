# Path to `1.0` — readiness tracker

This file is the authoritative roadmap from `0.x.y` alpha to `1.0`. For every component / pattern / token surface it records:

- **Status** — `stable` (API frozen for 1.0, no breaks in 1.x), `beta` (mostly done; specific open questions), `experimental` (public but may change before 1.0), or `internal` (don't rely on).
- **Open questions** — concrete decisions the maintainers need to resolve before promoting to `stable`.
- **Last reviewed** — date of the last sign-off pass.

Promotion rules:

1. A surface goes `experimental → beta` once the API shape is broadly settled and one real consumer has integrated it.
2. `beta → stable` requires (a) every open question resolved, (b) full Storybook + axe + visual coverage, (c) a contract doc in `docs/components/<name>.md`, (d) a per-component spec test suite that covers the documented behaviour.
3. Anything still `experimental` at `1.0` ships, but its package documentation must declare it as such; consumers know it can break in `1.1`.

`1.0` itself ships when **every** entry below is `stable` or explicitly deprecated.

## Token surface

| Area | Status | Notes |
|---|---|---|
| Color (primitive ramps) | `stable` | DTCG-sourced, frozen value set. Inversion strategy in dark is in tension with action shades — see open question. |
| Color (semantic) | `beta` | Names settled. Some component-specific tokens (`--kp-form-helper`, `--kp-button-bg`) still bypass the semantic layer; needs reconciliation. |
| Spacing / sizing | `stable` | 4 / 8 / 12 / 16 px scale, matches Figma. |
| Radius | `stable` | 8 / 10 / 12 / 14 / 16 px scale + `full` (`9999px`). |
| Typography | `beta` | Onest font + 11 / 12 / 13 / 14 / 16 / 20 px scale. Open: line-height across sizes. |
| Motion | `beta` | Duration tokens fixed; ease tokens under review. `prefers-reduced-motion` honored. |
| Elevation | `beta` | Three levels (`raised`, `floating`, `overlay`). Open: dark-mode shadow values. |
| Dark theme | `stable` | Three-layer architecture (primitive → semantic → theme override) with `color.text.*`, `color.surface.*`, `color.border.*` semantic families plus the `*.on-dark.*` invariant group for always-dark variants (`.kp-header--dark`, `.kp-sidebar--dark`). Lint-enforced — zero direct primitive references in component / pattern CSS. Both themes covered by visual regression and axe-core in CI. |

**Open token questions:**
- Should `--kp-color-white` carry both "surface elevation 0" and "high-contrast text on saturated bg" semantics? Currently double duty; in dark the surface meaning wins and the text meaning fails. Likely needs `color.fg.on-color` split. **Blocker for 1.0.**
- Are `gray-300 / gray-400` stable text colors across both themes, or do we need a semantic `color.text.muted` that swaps independently? **Blocker.**
- ~~Direct `var(--kp-color-blue-600)` references in component CSS (220 instances across 25 files) — audit and migrate to semantic accent tokens before 1.0.~~ **Resolved** (2026-05-05): full audit found 1334 direct primitive refs across 60 files (gray / brand / white). Migrated all of them to existing or new semantics (`color.text.*`, `color.surface.*`, `color.border.*`, `*.on-dark.*` invariants for always-dark variants). Verified by an in-tree count: zero primitive `var(--kp-color-{primitive})` references remain in component / pattern CSS.

## Component / feature gaps before 1.0

These are explicit "doesn't exist yet" entries — promotion to `1.0` may either ship without them (with a documented note) or require the gap closed first.

- ~~**Date range picker** — separate component; current `datepicker` is single-date only.~~ **Resolved** (2026-04-28): `kp-datepicker` already exposes `mode="range"` with hover-preview + min/max + presets. Description in the component table corrected.
- ~~**Virtualized list / table** — no built-in virtualization. Large datasets currently rely on external integration.~~ **Resolved** (2026-04-28): `@kanso-protocol/ui/virtual-list` ships fixed-height window-mode virtualization. `kp-table` users compose it around their row template; a baked-in `<kp-table-virtual>` is on the roadmap.
- ~~**File upload / dropzone** — not in catalog.~~ **Resolved** (2026-04-28): `@kanso-protocol/ui/file-upload` ships drag-and-drop + click-to-browse + managed file list (`setProgress` / `setStatus`).
- ~~**Standalone command palette** — exists as a variant of `search-bar`, not as its own component.~~ **Resolved** (2026-04-28): `@kanso-protocol/ui/command-palette` ships the dialog-backed component with full keyboard nav and a configurable global open-shortcut.
- ~~**Markdown viewer** — `rich-text-editor` only emits/edits HTML; no read-only markdown renderer.~~ **Resolved** (2026-04-28): `@kanso-protocol/ui/markdown-viewer` ships CommonMark + GFM via bundled `marked@7`; pluggable parser, three sizes, sanitized-by-default output.
- **Charts** — intentionally out of scope (use a charting lib of choice). Document the recommendation.

## Cross-cutting work needed before 1.0

1. ~~**Dark theme architecture stabilization**~~ **Resolved** (2026-05-05): three-layer DTCG architecture finalized; new `color.text.*` / `surface.*` / `border.*` semantic families with explicit dark overrides where primitive inversion was insufficient; `*.on-dark.*` invariant family for `.dark` pattern variants; zero direct primitive refs in component CSS.
2. ~~**i18n integration**~~ **Resolved** (2026-04-28): `@kanso-protocol/ui/i18n` ships `KP_LOCALE` / `KP_STRINGS` + Intl helpers. Migrated: datepicker, timepicker, file-upload, command-palette, search-bar. Remaining polish (folding `KP_VALIDATION_MESSAGES` into `KP_STRINGS`, auto 12h/24h detection) tracked in `docs/i18n.md`.
3. **RTL validation** — components use logical properties throughout, but no automated `dir="rtl"` test pass exists. Add a Playwright spec that visits each story under RTL and snapshots.
4. **Keyboard navigation audit** — most components implement WAI-ARIA patterns, but no centralized "keyboard map" doc lists the chord set per component.
5. ~~**Performance budget**~~ **Resolved** (2026-05-04): per-package gzipped fesm2022 budget recorded in `bundle-budget.json`; `npm run check:bundle-size` (also wired into CI `build` job) fails on regressions. Refresh with `--update` after intentional growth.
