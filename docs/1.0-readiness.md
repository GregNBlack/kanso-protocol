# Path to `1.0` â€” readiness tracker

This file is the authoritative roadmap from `0.x.y` alpha to `1.0`. For every component / pattern / token surface it records:

- **Status** â€” `stable` (API frozen for 1.0, no breaks in 1.x), `beta` (mostly done; specific open questions), `experimental` (public but may change before 1.0), or `internal` (don't rely on).
- **Open questions** â€” concrete decisions the maintainers need to resolve before promoting to `stable`.
- **Last reviewed** â€” date of the last sign-off pass.

Promotion rules:

1. A surface goes `experimental â†’ beta` once the API shape is broadly settled and one real consumer has integrated it.
2. `beta â†’ stable` requires (a) every open question resolved, (b) full Storybook + axe + visual coverage, (c) a contract doc in `docs/components/<name>.md`, (d) a per-component spec test suite that covers the documented behaviour.
3. Anything still `experimental` at `1.0` ships, but its package documentation must declare it as such; consumers know it can break in `1.1`.

`1.0` itself ships when **every** entry below is `stable` or explicitly deprecated.

## Token surface

| Area | Status | Notes |
|---|---|---|
| Color (primitive ramps) | `stable` | DTCG-sourced, frozen value set. Inversion strategy in dark is in tension with action shades â€” see open question. |
| Color (semantic) | `beta` | Names settled. Some component-specific tokens (`--kp-form-helper`, `--kp-button-bg`) still bypass the semantic layer; needs reconciliation. |
| Spacing / sizing | `stable` | 4 / 8 / 12 / 16 px scale, matches Figma. |
| Radius | `stable` | 8 / 10 / 12 / 14 / 16 px scale + `full` (`9999px`). |
| Typography | `beta` | Onest font + 11 / 12 / 13 / 14 / 16 / 20 px scale. Open: line-height across sizes. |
| Motion | `beta` | Duration tokens fixed; ease tokens under review. `prefers-reduced-motion` honored. |
| Elevation | `beta` | Three levels (`raised`, `floating`, `overlay`). Open: dark-mode shadow values. |
| Dark theme | `experimental` | Architecture under review. Designer-pass hardened the most-used token pairs but the primitive-inversion strategy fights itself for action shades. Needs `color.accent.*` semantic family + audit of direct primitive references in components. |

**Open token questions:**
- Should `--kp-color-white` carry both "surface elevation 0" and "high-contrast text on saturated bg" semantics? Currently double duty; in dark the surface meaning wins and the text meaning fails. Likely needs `color.fg.on-color` split. **Blocker for 1.0.**
- Are `gray-300 / gray-400` stable text colors across both themes, or do we need a semantic `color.text.muted` that swaps independently? **Blocker.**
- Direct `var(--kp-color-blue-600)` references in component CSS (220 instances across 25 files) â€” audit and migrate to semantic accent tokens before 1.0.

## Cross-cutting work needed before 1.0

1. **Dark theme architecture stabilization** â€” see open token questions.
2. **i18n integration** â€” date/time pickers, validation messages, ARIA labels. Currently English-only. Consider an `injectionToken`-based approach matching `KP_VALIDATION_MESSAGES`.
3. **RTL validation** â€” components use logical properties throughout, but no automated `dir="rtl"` test pass exists. Add a Playwright spec that visits each story under RTL and snapshots.
4. **Keyboard navigation audit** â€” most components implement WAI-ARIA patterns, but no centralized "keyboard map" doc lists the chord set per component.
5. **Performance budget** â€” bundle-size budget per package; current sizes are unaudited.
